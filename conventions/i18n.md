# i18n Conventions for swift-toolkit

> Date: 2026-04-27. Companion to `docs/superpowers/plans/2026-04-27-i18n-localization.md`.

## Goal

swift-toolkit is single-source-of-truth in English. User-facing strings are localized per user's chosen language. Currently supported: `en`, `ru`.

## Where strings live

| String type                                  | File                                                     | Language |
|----------------------------------------------|----------------------------------------------------------|----------|
| Skill body (algorithm, structure, contracts) | `skills/<name>/SKILL.md`                                 | EN only  |
| Skill user-facing strings                    | `skills/<name>/locales/<lang>.md`                        | per lang |
| Agent body                                   | `agents/<name>.md`                                       | EN only  |
| Agent description (frontmatter)              | `agents/<name>.md` frontmatter                           | bilingual |
| Command description (frontmatter)            | `commands/<name>.md` frontmatter                         | bilingual one-line |
| Command body                                 | `commands/<name>.md` body                                | EN only  |
| CLAUDE.md template (copied to user projects) | `templates/claude-md/<lang>.md`                          | per lang |
| README                                       | `README.md` (EN); `docs/readme.ru.md` (optional mirror)  | EN main, RU mirror |

## What is user-facing

(Same definitions as in the plan's Conventions section. Reproduced here so this spec is self-contained.)

A string is **user-facing** (localized) if any of these hold:
1. Printed to chat as Claude's reply.
2. Used as `AskUserQuestion` text/header/option label.

A string is **internal** (always EN) if any of these hold:
- It's a stable identifier (`name:` field, contract field, status enum value).
- It's a file path, glob, command name, env var.
- It documents algorithm steps inside the skill body.
- **It's a section heading or structural label written into a Markdown file by the toolkit** (`Task.md`, `Done.md`, `Plan.md`, `Research.md`, `ChangesRequested.md`, `Review.md`, etc.). File templates are stable EN structure so downstream skills can parse them and so files don't break when `LANG` is switched mid-project.

Free-form **content** the user composes inside an artifact (prose under a heading, notes under bullet points) follows the user's language naturally — not a localized template, just user input.

## Locale file format

`skills/<name>/locales/en.md` and `ru.md`. Both MUST have identical key sets.

Example (`skills/orchestrator/locales/en.md`):

```markdown
# orchestrator — en

## error_no_task_id
Specify task number, e.g. `/task-run 026`.

## confirm_dispatch
Profile: {profile}, mode: {mode}, stack: {stack}, start: {start_stage}. Correct?
```

Keys are `lower_snake_case`. Placeholders use `{name}`.

## Language resolution (every localized skill includes this section verbatim)

1. Read `CLAUDE.md` from project root.
2. Find `## Language` section.
3. First non-empty line, lowercased and trimmed = `<lang>`.
4. If `<lang>` ∈ {`en`, `ru`} → use it. Otherwise default to `en`.
5. Read `locales/<lang>.md` of the calling skill. Look up by H2.
6. Missing key → fallback to `en`. Still missing → bug, fail loudly.

Cache `<lang>` once per skill invocation.

## Bilingual triggers

Skill / agent description:

```yaml
description: |
  <EN summary>.
  Use when (en): "<trigger 1>", "<trigger 2>"
  Use when (ru): "<триггер 1>", "<триггер 2>"
```

Command description (one-line):

```yaml
description: "<EN summary> / <RU summary>"
```

## How to add a new language

1. For each localized skill, copy `locales/en.md` to `locales/<new>.md` and translate values (keep keys).
2. Add `<new>` to the supported list in `swift-setup`'s LANG question and in the language resolution allowlist (if added there).
3. Create `templates/claude-md/<new>.md` (copy from `en.md`, translate persona block).
4. Optionally extend `description:` triggers to include `Use when (<new>):` lines for the most common patterns.

## CLAUDE.md schema (after Phase 7)

Top of the project CLAUDE.md will look like this:

```markdown
# CLAUDE.md — <Project Name>

> Project-level config for a Swift/Apple project.

## Language

en

## Persona

(localized persona text — see templates/claude-md/<lang>.md)

## Stack

- UI: ...
- Async: ...
- DI: ...
- Architecture: ...
- Platform: ...
- Tests: ...

## Mode

manual

## Modules

(optional)

## Paths

(optional)

## Orchestration

(localized orchestration overview)
```

The `## Language` section MUST be the first H2 in the file (after the H1 title and any blockquote intro). Section name is fixed (English `Language`, never translated) so the resolution procedure has a stable anchor.
